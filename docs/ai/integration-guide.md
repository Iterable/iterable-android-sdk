# Golden Path: Iterable Android SDK Integration

A step-by-step guide for an AI agent to integrate the Iterable SDK into any Android app. Contains critical gotchas and traps discovered through real integration failures.

> **Agent note:** Throughout this guide, placeholders like `YourApplication`, `your.package.name`, and `yourscheme` must be replaced with values from the user's actual project. When in doubt, ask the user.

---

## Setup Mode

Before starting, ask the developer which setup mode they prefer:

- **Quick setup** — Ask only 🔴 Critical questions. Use sensible defaults for everything else. Gets to a working SDK fast.
- **Guided setup** — Ask 🔴 Critical + 🟠 Important questions. Default the rest. Recommended for most integrations.
- **Full control** — Ask 🔴 Critical + 🟠 Important + 🟡 Relevant questions. Only 🟢 Optional decisions are made by the agent.

If the developer doesn't have a preference, use **Guided setup**.

### Decision tier legend

- 🔴 **Critical** — Must ask. Cannot proceed without a human answer.
- 🟠 **Important** — Should ask. Has real impact, but the agent can suggest a default if the developer defers.
- 🟡 **Relevant** — Better if decided by a human, but a sensible default exists and won't cause problems.
- 🟢 **Optional** — Agent can pick the best option and move on silently.

## Build System Compatibility

> **Agent note — Groovy vs Kotlin DSL:** All code examples in this guide use Kotlin DSL (`.gradle.kts`). If the project uses Groovy (`.gradle` files), you MUST translate:
> - `val` → `def`
> - Remove `=` from property assignments (`buildConfig true` not `buildConfig = true`)
> - `import` statements must be at file top level, not inside blocks
> - `implementation("group:artifact:version")` → `implementation 'group:artifact:version'`
> - String templates with nested quotes may need different escaping
>
> Check the project's build files to determine which syntax is in use before writing any Gradle code.

> **Agent note — Multi-module projects:** If the project has multiple modules, add SDK dependencies and the `google-services` plugin to the **app module** (the one with `com.android.application` plugin), not the root project or library modules. The app module may not be named `app/` — check `settings.gradle` to find it.

## Prerequisites

Before writing any code, collect information from the developer. Which questions to ask depends on the chosen setup mode.

### 🔴 Critical (always ask)

1. **Iterable API key** — Must be a Mobile (client-side) key. Never a Server-side key.
2. **Is JWT required?** — If yes, the developer needs a server-side JWT generator. Without an auth handler, a JWT-required key causes ALL SDK calls to silently fail with zero errors.
3. **Firebase `google-services.json`** — From Firebase Console. Required for push notifications.
4. **The user's email or userId** — How does the app identify logged-in users? This determines whether to call `setEmail()` or `setUserId()`.

### 🟠 Important (ask in Guided and Full control)

5. **Push integration name** — What it's called in the Iterable dashboard. Often the app package name, but must match exactly. Default: app package name.
6. **Custom deep link scheme** — e.g., `myapp://`. If the app uses one, it must be registered with `setAllowedProtocols`. Default: none (HTTPS only).
7. **Which features to enable** — Push, in-app messages, mobile inbox, embedded messaging, deep linking? Determines which per-feature guides to load.

### 🟡 Relevant (ask in Full control)

8. **Embedded message placement IDs** — Auto-generated by Iterable dashboard. Required only if using embedded messaging.
9. **Data region** — US (default) or EU. EU customers must set `setDataRegion(IterableDataRegion.EU)`.
10. **Notification icon and accent color** — Brand/design decision. Uses app icon by default but should be a proper silhouette icon.

### 🟢 Optional (agent decides)

11. **Auto push registration** — Default: `true`. Almost always correct.
12. **In-app display interval** — Default: 30 seconds between auto-shown messages.
13. **In-app storage type** — Default: file-based. Use in-memory only for strict privacy requirements.

## Step 1: Dependencies

> **Agent note — version resolution:** Do NOT use hardcoded versions. Before adding any dependency:
> - **Iterable SDK**: Check `CHANGELOG.md` in this repo — first `## [x.y.z]` entry after `## [Unreleased]` is the latest.
> - **Firebase BOM**: Look up the latest version from [Firebase Android releases](https://firebase.google.com/support/release-notes/android) or Maven Central.
> - **Google Services plugin**: Look up the latest from Maven Central (`com.google.gms:google-services`).

**`build.gradle.kts` (project-level)**:
```kotlin
plugins {
    id("com.google.gms.google-services") version "<latest>" apply false
}
```

**`app/build.gradle.kts`** (Kotlin DSL):
```kotlin
plugins {
    id("com.google.gms.google-services")
}

import java.util.Properties
import java.io.FileInputStream
val localProperties = Properties()
val localPropertiesFile = rootProject.file("local.properties")
if (localPropertiesFile.exists()) {
    localProperties.load(FileInputStream(localPropertiesFile))
}

android {
    defaultConfig {
        buildConfigField("String", "ITERABLE_API_KEY", "\"${localProperties.getProperty("ITERABLE_API_KEY", "")}\"")
    }
    buildFeatures {
        buildConfig = true
    }
}

dependencies {
    implementation("com.iterable:iterableapi:<latest>")
    implementation("com.iterable:iterableapi-ui:<latest>")
    implementation(platform("com.google.firebase:firebase-bom:<latest>"))
    implementation("com.google.firebase:firebase-messaging")
}
```

**`app/build.gradle`** (Groovy — if the project uses `.gradle` files):
```groovy
plugins {
    id 'com.google.gms.google-services'
}

// At file top level, before android { }
import java.util.Properties
import java.io.FileInputStream

android {
    defaultConfig {
        def localProperties = new Properties()
        def localPropertiesFile = rootProject.file("local.properties")
        if (localPropertiesFile.exists()) {
            localProperties.load(new FileInputStream(localPropertiesFile))
        }
        buildConfigField "String", "ITERABLE_API_KEY", "\"${localProperties.getProperty("ITERABLE_API_KEY", "")}\""
    }
    buildFeatures {
        buildConfig true  // no '=' in Groovy
    }
}

dependencies {
    implementation 'com.iterable:iterableapi:<latest>'
    implementation 'com.iterable:iterableapi-ui:<latest>'
    implementation platform('com.google.firebase:firebase-bom:<latest>')
    implementation 'com.google.firebase:firebase-messaging'
}
```

**`local.properties`** (gitignored):
```
ITERABLE_API_KEY=your_key_here
```

Place `google-services.json` in `app/`.

## Step 2: SDK Initialization (Application class)

> **Agent note:** Replace `YourApplication` with the app's actual Application class. Omit config lines for features the developer said they don't need.

```kotlin
class YourApplication : Application() {
    override fun onCreate() {
        super.onCreate()

        val config = IterableConfig.Builder()
            .setAutoPushRegistration(true)
            .setPushIntegrationName("your.package.name")        // 🟠 Must match Iterable dashboard exactly
            .setInAppHandler(AppInAppHandler())                  // See in-app-messages.md
            .setCustomActionHandler(AppCustomActionHandler())    // See deep-linking.md (only if using custom actions)
            .setUrlHandler(AppUrlHandler(this))                  // See deep-linking.md
            .setAllowedProtocols(arrayOf("yourscheme"))          // 🟠 Only if using custom deep link schemes
            .setAuthHandler(AppAuthHandler { generateJwtForEmail(getFreshEmail()) })  // 🔴 Only if JWT required — see jwt-authentication.md
            .setEnableEmbeddedMessaging(true)                    // 🟠 Only if using embedded messages
            .build()

        IterableApi.initializeInBackground(this, BuildConfig.ITERABLE_API_KEY, config) {
            Log.d("Iterable", "SDK initialized")
        }
    }
}
```

### Gotcha: `initializeInBackground` takes 4 args
`initializeInBackground(context, apiKey, config, callback)` — the callback is the 4th arg.

### Gotcha: Do NOT call setEmail in the init callback
The SDK's `IterableAuthManager` has internal retry limits. Calling `setEmail` too early consumes the retry budget, causing `AUTH_TOKEN_MISSING` for all subsequent calls in the same process.

### Gotcha: Always wrap SDK calls in `onSDKInitialized`
When using `initializeInBackground`, wrap SDK calls in `IterableApi.onSDKInitialized { }` to ensure the SDK is ready before proceeding.

## Step 3: User Identification

> **Agent note:** Adapt this to the app's login flow. The 🔴 Critical prerequisite determines whether to use `setEmail` or `setUserId`.
>
> **Apps with no login system:** If the app has no user authentication, generate a persistent device UUID (stored in SharedPreferences) and use `setUserId(uuid)`. Call this once during app startup after SDK initialization — NOT inside the `initializeInBackground` callback (see Gotcha #4). Example:
> ```kotlin
> // In your Application class or main Activity, OUTSIDE the init callback:
> IterableApi.onSDKInitialized {
>     val userId = getOrCreateDeviceUuid()  // read/generate from SharedPreferences
>     IterableApi.getInstance().setUserId(userId)
> }
> ```
> If the app later acquires an email (e.g., through a purchase or settings screen), switch to `setEmail(email)` at that point.

```kotlin
// On app restart (user already logged in):
if (isUserLoggedIn()) {
    IterableApi.onSDKInitialized {
        IterableApi.getInstance().setEmail(getSavedEmail())
    }
    navigateToMainScreen()
    return
}

// On fresh login — use setEmail's success callback to chain dependent calls:
IterableApi.onSDKInitialized {
    IterableApi.getInstance().setEmail(
        email,
        { // onSuccess — JWT auth is complete, safe to call updateUser now
            val dataFields = JSONObject()
            dataFields.put("name", name)
            IterableApi.getInstance().updateUser(dataFields)
        },
        { reason, _ -> // onFailure
            Log.e("Iterable", "setEmail failed: $reason")
        }
    )
}
```

> **Agent note:** `setEmail` has multiple overloads. Use the one with `SuccessHandler` and `FailureHandler` callbacks: `setEmail(email, onSuccess, onFailure)`. This ensures dependent calls like `updateUser` only fire after JWT authentication completes. Never use `Handler.postDelayed` or arbitrary delays to sequence SDK calls.

### Gotcha: Don't call registerForPush() explicitly
`setAutoPushRegistration(true)` handles it automatically when `setEmail` is called.

## Top 15 Traps (ordered by severity)

1. **JWT-required key with no auth handler** → ALL SDK calls silently fail, zero errors
2. **JWT secret encoding** → use raw UTF-8, not hex-decode, not base64-decode
3. **Caching email at startup for the auth handler** → `AUTH_TOKEN_NULL`. The auth handler lambda must read the email fresh from the source of truth (DataStore, SharedPreferences, etc.) every time `onAuthTokenRequested()` is called
4. **Calling setEmail/setUserId in Application.onCreate init callback** → poisons SDK auth retry state. Use `IterableApi.onSDKInitialized { }` OUTSIDE the init callback instead
5. **Missing runtime POST_NOTIFICATIONS permission** → push silently fails on Android 13+. Must use `ActivityResultContracts.RequestPermission()` declared as a **class property** on the Activity (NOT inside a function body — `registerForActivityResult` throws `IllegalStateException` if called dynamically)
6. **`InAppResponse` is a nested enum** inside `IterableInAppHandler` → using bare `InAppResponse` without importing `IterableInAppHandler.InAppResponse` causes "Unresolved reference" compile errors
7. **Custom URL schemes silently blocked** → must use `setAllowedProtocols`
8. **Groovy vs Kotlin DSL in build files** → pasting Kotlin DSL syntax (`val`, `=` assignments, `import` inside blocks) into a Groovy `.gradle` file causes build failures. Check the project's build files first
9. **`google-services.json` must match the running package name** → if the app uses `applicationIdSuffix ".debug"`, the Firebase project must include BOTH the release and debug package names or the build fails
10. **`onTokenRegistrationSuccessful` is local, not server** → don't trust it as server confirmation
11. **Firing setEmail + updateUser + registerForPush simultaneously** → auth race condition. Use `setEmail(email, onSuccess, onFailure)` and chain `updateUser` in `onSuccess`
12. **Accessing embeddedManager before SDK init** → crash
13. **Placement IDs are auto-generated** → never hardcode assumed values
14. **Message Types must exist before templates** → dashboard prerequisite for every channel
15. **Don't hallucinate Iterable dashboard paths** → defer to user or official docs
